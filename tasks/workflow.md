# 4-Agent 长任务续跑工作流

## 角色分工

### main
- 接收用户任务
- 判断是否是长任务
- 创建 / 更新任务记录
- 指派执行者
- 对用户做最终汇总

### monkey
- 内容生产执行
- 适合：文案、PPT、选题、改稿、结构化写作

### panda
- 信息搜集执行
- 适合：新闻、天气、资料搜索、外部信息整理

### assi
- 监督 agent
- 适合：检查任务是否停住、推动继续、做阶段汇报

## 长任务判定

满足任一条就按长任务处理：

- 预计超过 15 分钟
- 需要多轮执行
- 一次回复通常做不完
- 需要“搜集 → 整理 → 输出”多个阶段

## 执行规则

1. 长任务先登记，再执行。
2. owner agent 每轮只推进一个小阶段。
3. 每轮结束后必须更新：
   - `status`
   - `current_step`
   - `last_progress_at`
   - `next_action`
   - `summary`
4. 如果任务未完成，不要静默消失。
5. 如果卡在需要用户拍板，把 `needs_user_input` 设为 `true`，并写明原因。
6. `assi` 只盯活跃任务，不盯已完成任务。

## assi 的监督标准（第一版）

当以下条件同时满足时，可视为“疑似停住”：

- `status` 是 `running`
- `needs_user_input` 不是 `true`
- 距离 `last_progress_at` 已超过一段时间（建议 45 分钟）

此时 `assi` 应做两件事：

1. 给用户一句简短阶段汇报
2. 推动 owner agent 继续下一阶段

## 阶段汇报格式

推荐短格式：

- 已完成：<一句>
- 当前：<一句>
- 下一步：<一句>

例：

- 已完成：PPT 目录已出。
- 当前：还在补逐页文案。
- 下一步：继续写前 5 页并统一语气。

## 暂停条件

出现以下任一情况，暂不强推续跑：

- `status = done`
- `status = paused`
- `status = blocked`
- `needs_user_input = true`

## 设计原则

- 先稳，再自动化。
- 先有任务状态，再谈监督。
- 先人工可读，再追求花哨。
- 不要让 4 个 agent 同时抢一个任务。
